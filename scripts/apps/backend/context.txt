SECTION 1: WHAT IS DUSTWATCH (THE ELEVATOR PITCH)
================================================================================

DustWatch is a satellite-powered AI platform that detects active construction
sites in Indian cities using free satellite imagery, scores them by pollution
risk, and makes the evidence publicly accessible.

The problem it solves:
  - Construction activity contributes roughly 30% of urban PM10 dust pollution
    in Indian cities.
  - Most construction operates with self-reported compliance. Many sites have
    no building permit at all — they are invisible to municipal authorities.
  - There is no independent, scalable way to verify which construction sites
    exist, how big they are, or whether they follow dust suppression rules.

What DustWatch does:
  1. Downloads satellite imagery from ESA's Sentinel-1 (radar) and Sentinel-2
     (optical) satellites via Google Earth Engine — completely free data.
  2. Runs change detection algorithms to find areas where vegetation was
     removed and bare soil appeared (a signature of construction activity).
  3. Feeds those candidate zones into a Random Forest machine learning model
     that classifies them as "construction" or "not construction."
  4. Cross-references detected sites against building permit records to flag
     unregistered/illegal construction.
  5. Computes a composite risk score for each site based on area, duration,
     proximity to schools/hospitals, and permit status.
  6. Serves all this through a REST API that powers a map-based web frontend
     where anyone can browse, filter, and share detected violations.

Who would use it:
  - Urban residents concerned about dust near their homes/schools.
  - Journalists investigating illegal construction.
  - NGOs filing PILs (Public Interest Litigations) against polluters.
  - Municipal authorities for independent verification of permit compliance.

What makes it unique:
  - Every detection is traceable to publicly available satellite imagery.
    Anyone can independently verify a detection by checking the same Sentinel
    images on Google Earth Engine.
  - It does NOT claim to measure air quality. It identifies dust SOURCES
    (construction sites) and estimates their contribution using physics-based
    dispersion modeling.

SECTION 2: THE TECH STACK
================================================================================

Backend (Python 3.10+):
  - FastAPI + Uvicorn — REST API framework
  - Google Earth Engine (earthengine-api, geemap) — satellite data access
  - scikit-learn — Random Forest classifier
  - rasterio — GeoTIFF raster processing
  - geopandas + shapely — geospatial polygon operations
  - numpy + pandas — data processing
  - Pydantic — API data validation
  - python-dotenv — environment variable management
  - requests — HTTP client for weather/OSM APIs
  - joblib — model serialization
  - pyproj — coordinate system transformations
  - matplotlib + Pillow — dust dispersion heatmap rendering

Frontend (TypeScript):
  - React 19 + TypeScript 6
  - Vite 8 — build tool and dev server
  - Leaflet.js + react-leaflet — interactive map
  - Axios — HTTP client for API calls
  - Tailwind CSS — styling (installed but lightly used; most styling is inline)
  - date-fns — date formatting

Data Sources:
  - Sentinel-2 SR Harmonized (Copernicus/ESA) — 10m optical imagery
  - Sentinel-1 GRD (Copernicus/ESA) — radar imagery (works through clouds)
  - OpenStreetMap (Nominatim API) — school/hospital locations
  - Open-Meteo Archive API — historical wind speed/direction data
  - VMC permit data — placeholder/sample only (20 fake permits)

--- data-pipeline/ (STANDALONE ML PIPELINE) ---

This folder contains the original "from-scratch" pipeline that was built first.
It runs as a sequence of standalone Python scripts, completely separate from the
backend web app. It works directly with local GeoTIFF raster files.

  config.py:
    Central configuration for the standalone pipeline. Defines:
    - All file paths (data dirs, model paths, output paths)
    - GEE project ID and AOI bounding box for Vadodara
    - Imagery date ranges (T1: Jan-Apr 2025 "before", T2: Oct 2025-Jan 2026 "after")
    - Patch extraction parameters (256x256 pixels, stride 128)
    - Band names for the 8-band change detection stack
    - ensure_dirs() helper to create output directories

  change_detection.py:
    Connects to Google Earth Engine and builds an 8-band change detection
    GeoTIFF. Computes:
    - Sentinel-2 median composites for "before" and "after" periods
    - NDVI (vegetation index) for both periods
    - BSI (bare soil index) for both periods
    - DELTA_NDVI and DELTA_BSI (temporal differences)
    - Sentinel-1 SAR VV and VH bands (radar backscatter)
    Exports the stacked image to Google Drive as a GeoTIFF.
    This is the FIRST script you run. Takes 10-30 min on GEE.

  extract_patches.py:
    Takes the downloaded GeoTIFF and slices it into 256x256 pixel patches
    with 128-pixel overlap (stride). Each patch is saved as a .npy numpy file.
    Also creates a patches_index.json ranking patches by "construction likelihood"
    (high BSI change + negative NDVI change = more likely construction).
    Currently produces 725 patches from the Vadodara raster.

  create_starter_labels.py:
    Generates weak/heuristic labels automatically. Takes the top 100 patches
    from patches_index (by priority score) as positive labels (likely
    construction) and the bottom 100 as negative labels (likely not construction).
    Saves to labels.json. These are WEAK labels — they're not manually verified.

  suggest_patches.py:
    An active learning helper. Reads the patches index, filters out already-
    labeled patches, and suggests up to 400 new patches to label manually.
    Uses heuristics: high BSI + NDVI drop = suggest as positive; NDVI increase
    = suggest as negative; known river/road areas = hard negatives.
    Saves suggestions to labeling_queue.json.
    NOTE: References config.LABELING_QUEUE_JSON which doesn't exist in config.py.
    This script would crash if run. It's incomplete.

  train_model.py:
    Trains a Random Forest classifier on the labeled patches. Loads patches
    from .npy files, extracts mean band values as features (8 features per
    patch), splits train/test, trains with 200 estimators, evaluates, and
    saves the model + feature importance rankings.

  generate_geojson.py:
    The final inference step. Loads the trained model, scans the entire GeoTIFF
    with a sliding window, classifies each patch, builds a binary prediction
    map, vectorizes it into polygons using rasterio.features.shapes, merges
    overlapping polygons with shapely.unary_union, and saves as GeoJSON.

  requirements.txt:
    Dependencies: numpy, rasterio, shapely, scikit-learn, joblib, python-dotenv,
    earthengine-api. No version pins for most packages.


--- scripts/apps/backend/ (FASTAPI WEB APPLICATION) ---

This is the SECOND, more complete version of the pipeline that also includes
a web server. It duplicates and extends the data-pipeline functionality.

  main.py:
    FastAPI application entry point. Sets up:
    - CORS middleware allowing localhost:3000 and localhost:5173
    - Static file serving from /static directory
    - Three router modules mounted at /api prefix
    - Root endpoint (/) returning version info
    - Health check endpoint (/health)

  config.py:
    Same as data-pipeline/config.py but with paths adjusted for the
    scripts/apps/backend/ location. ROOT_DIR goes up 3 levels instead of 1.
    Loads .env from both project root and backend directory.
    RAISES ValueError if GEE_PROJECT_ID is not set — the backend will not
    start without it, even if you're not using GEE features.

  requirements.txt:
    FastAPI, uvicorn, earthengine-api, geemap, geopandas, shapely,
    scikit-learn, numpy, rasterio, pydantic, python-dotenv, pandas, joblib.
    No version pins.

  models/site.py:
    Pydantic data models that define the API response shapes:
    - ConstructionSite: id, name, coordinates, polygon, riskLevel, riskScore,
      permitStatus, areaM2, activeDays, nearbySchools, nearbyHospitals,
      detectedAt, lastUpdated, satelliteImageBefore/After, ward, address
    - CityStats: totalSites, critical/high/medium/low counts, unregistered
      count, populationExposed, lastUpdated
    These schemas MUST match the frontend TypeScript types exactly.

  routers/sites.py:
    API endpoints for construction site data:
    - GET /api/sites → returns all sites from vadodara_sites.geojson
    - GET /api/sites/{site_id} → returns a single site by ID
    Reads from data/samples/vadodara_sites.geojson (relative path).
    Returns 404 if the file doesn't exist (pipeline hasn't been run yet).

  routers/stats.py:
    API endpoint for city-level statistics:
    - GET /api/stats → aggregates all sites into counts by risk level,
      permit status, and estimated population exposed.
      Calls compute_city_stats() from risk_scorer service.
      Returns zero-stats if the GeoJSON file doesn't exist.
    - GET /api/stats/population_exposure → calculates estimated residents exposed
      to modeled dust plumes across 4 zones in Vadodara (Voronoi zones).

  routers/layers.py:
    API endpoints for the dust dispersion overlay layer:
    - GET /api/layers/modeled_dust?period=...&pollutant=... → returns the URL
      of a pre-rendered PNG heatmap, map bounds, wind data, and legend info.
    - GET /api/layers/available_periods → lists the 3 time periods available.
    - GET /api/layers/aqi_stations → returns real-time AQI measurement stations
      for Vadodara using the WAQI (World Air Quality Index) API. Uses WAQI_TOKEN.
    Validates period (oct_dec_2025, jan_mar_2026, apr_may_2026) and pollutant
    (pm10, pm25). The PNGs must be pre-generated by build_dust_dispersion.py.

  services/gee_client.py:
    Google Earth Engine helper functions:
    - Initializes GEE connection on import (side effect)
    - get_sentinel2_composite() — fetches S2 imagery, tries cloud thresholds
      of 20%, 50%, 80% progressively, computes NDVI, NDBI, BSI
    - get_sentinel1_composite() — fetches S1 SAR mean VV/VH
    - get_combined_composite() — stacks S2 + S1
    - get_monthly_composites() — builds one S2 composite per month
    - get_monthly_sar_composites() — builds one S1 composite per month

  services/change_detection.py:
    The core detection logic. Two approaches:
    1. detect_construction_candidates() — simple bi-temporal comparison:
       BSI increase > 0.08, NDVI decrease > 0.05, after BSI > 0.02,
       after NDVI < 0.25, SAR VV > -16. Morphological cleanup, vectorize,
       filter by area (2000-30000 m²).
    2. compute_temporal_features() — advanced 21-band temporal feature stack
       from 3 monthly composites. Includes BSI trends, NDVI trends, NDBI
       (built-up index), MNDWI (water index to filter rivers), SAR trends,
       VH/VV ratio change, and VV-VH double-bounce proxy.

  services/classifier.py:
    ML model management:
    - 21 temporal feature names defined
    - extract_features() — uses GEE reduceRegions to sample temporal image
      at candidate polygon centroids
    - train_classifier() — trains RF with 200 trees, max_depth 15
    - load_classifier() — loads from .pkl, tries two file paths
    - run_inference() — runs model.predict_proba, handles shape mismatches
      by padding with zeros, returns confidence scores
    - filter_by_confidence() — filters candidates above a threshold

  services/risk_scorer.py:
    Composite risk scoring algorithm:
    - Area factor: area_m2 / 20000, capped at 1.0
    - Duration factor: active_days / 90, capped at 1.0
    - Proximity factor: (schools*0.6 + hospitals*0.4) / 5, capped at 1.0
    - Permit factor: registered=0, expired=0.6, unregistered=1.0
    - Weights: area 25%, duration 25%, proximity 30%, permit 20%
    - Score = weighted sum × 100 → risk level (critical ≥75, high ≥50,
      medium ≥25, low <25)
    - compute_city_stats() — counts sites by risk level, sums population
      exposed using heuristic formula (areaM2 × 0.012 for critical/high sites)

  services/permit_lookup.py:
    Cross-references detected site polygons against permit geometries:
    - load_permits() — reads vmc_permits.json
    - get_permit_status() — checks if site polygon intersects any permit polygon
    - tag_sites_with_permits() — tags all sites with permit status
    Currently uses 20 permit entries. Almost every site gets tagged
    as "unregistered" because the permit data is not real.

  services/dust_dispersion.py:
    Gaussian plume air dispersion model:
    - Computes PM10/PM2.5 concentration grids at 50m resolution
    - Uses EPA AP-42 emission factors for construction dust
    - Briggs urban sigma parameterization for plume spread
    - Builds concentration from all construction sites with given wind
    - Saves as GeoTIFF and semi-transparent PNG overlay
    - PNG uses viridis colormap with log-scale normalization

  services/weather_client.py:
    Fetches historical wind data from Open-Meteo Archive API:
    - 3 predefined periods (Oct-Dec 2025, Jan-Mar 2026, Apr-May 2026)
    - Computes vector-averaged wind speed and direction
    - Caches results as JSON files to avoid repeated API calls
    - Used by dust dispersion model

  services/population_exposure.py:
    Zone-level population exposure calculation:
    - Defines 4 geographical zones of Vadodara (North, East, West, South) with
      wards, population (2011 Census), area, and centroids.
    - Uses Voronoi partitioning (nearest centroid by Euclidean distance) to assign
      each grid cell of the dust dispersion GeoTIFF to a zone.
    - Computes population exposure by multiplying the exposed area (where dust
      concentration is above a threshold) with the zone's population density.

  scripts/gee_pipeline.py:
    THE MAIN PIPELINE ORCHESTRATOR. Run this to generate all detections.
    Execution flow:
    1. Initialize GEE
    2. Run change detection (bi-temporal, saves candidates.geojson)
    3. Build monthly S2/S1 composites (Oct/Nov/Dec 2025)
    4. Compute 21-band temporal feature image
    5. Extract features for each candidate zone
    6. Run Random Forest inference
    7. Filter by confidence threshold (0.45)
    8. Build site records with IDs, coordinates, areas
    9. Tag permit status
    10. Compute risk scores
    11. Filter by area (1000-50000 m²)
    12. Write output to data/samples/vadodara_sites.geojson
    Takes 5-15 minutes depending on GEE load.

  scripts/train_model.py:
    SIMPLE SYNTHETIC DATA TRAINER. Generates fake training data with
    hardcoded statistical distributions for 5 features (not 21!).
    Construction class: low NDVI, high BSI, specific SAR ranges.
    Non-construction: opposite distributions.
    Trains a basic RF model. This was used as a placeholder before real
    training data was available.
    BUG: Line 6 imports numpy as "npe" instead of "np" — this script will
    crash if run.

  scripts/build_real_training_data.py:
    Builds REAL training data by sampling GEE imagery at labeled coordinates.
    Uses a CSV of positive examples (manually verified construction sites)
    and the pipeline's false positive outputs as negatives.
    Samples all 21 temporal feature bands at each point.
    Saves as vadodara_real_training_data.csv.

  scripts/retrain_with_labels.py:
    Retrains the RF classifier on real labeled data (from build_real_training_data).
    Key improvements over the simple trainer:
    - Uses all 21 temporal features
    - Clips VH/VV ratio outliers
    - Uses class_weight="balanced" for imbalanced data
    - 400 estimators, max_depth 10
    - Runs 5-fold cross-validation
    - Prints feature importance rankings
    - Saves to both primary and secondary model paths

  scripts/enrich_proximity.py:
    Enriches detected sites with proximity data:
    - Contains hardcoded coordinates for 15 known Vadodara schools and
      14 known hospitals (manually verified from OSM/Google Maps)
    - Queries Nominatim (OpenStreetMap) API for additional school/hospital
      locations
    - Counts how many schools/hospitals are within 500m of each site
    - Re-runs risk scoring with updated proximity data
    - Writes enriched GeoJSON back to the same file

  scripts/build_dust_dispersion.py:
    Generates dust dispersion heatmap images:
    - Loads detected sites from GeoJSON
    - For each of 3 time periods, fetches wind data from weather_client
    - Runs Gaussian plume model for both PM10 and PM2.5
    - Saves GeoTIFF rasters and transparent PNG overlays
    - PNGs are served as static files through the /static endpoint

  scripts/diagnose_gee.py:
    Diagnostic script for troubleshooting GEE connectivity:
    - Counts available Sentinel-2 images at various cloud thresholds
    - Shows monthly image counts for Oct/Nov/Dec 2025
    - Computes BSI statistics for T1 and T2 periods
    - Useful for debugging "no images found" errors

  scripts/create_starter_labels.py:
    Duplicate of data-pipeline/create_starter_labels.py. Same functionality.

  scripts/extract_patches.py:
    Duplicate of data-pipeline/extract_patches.py. Same functionality.

================================================================================
SECTION 5: HOW THE DATA FLOWS (END TO END)
================================================================================

Step 1: Satellite Imagery Acquisition
  - GEE downloads Sentinel-2 optical and Sentinel-1 radar imagery for Vadodara
  - "Before" period: Jan-Apr 2025 (baseline)
  - "After" period: Oct-Dec 2025 (construction activity period)
  - Cloud-filtered, median-composited

Step 2: Change Detection
  - Computes spectral indices (NDVI, BSI, NDBI) for both time periods
  - Calculates temporal differences (DELTA_NDVI, DELTA_BSI)
  - Thresholds: BSI increase + NDVI decrease = likely construction
  - Morphological cleanup to remove noise
  - Vectorizes binary mask into polygons
  - Filters by area (2000-30000 m²)

Step 3: Feature Extraction
  - Builds 21-band temporal feature stack from 3 monthly composites
  - Samples feature values at each candidate polygon
  - Features include BSI trends, NDVI trends, NDBI, SAR backscatter, etc.

Step 4: Classification
  - Random Forest model classifies each candidate
  - Confidence threshold: 0.45 (candidates below this are dropped)
  - Model was trained on ~200 labeled examples (67 positives, 134 negatives)

Step 5: Enrichment
  - Permit cross-referencing against VMC data (currently fake)
  - Proximity computation to schools/hospitals via OSM
  - Risk scoring (area × 0.25 + duration × 0.25 + proximity × 0.30 + permit × 0.20)
  - Dust dispersion modeling (Gaussian plume with real wind data)

Step 6: API Serving
  - FastAPI reads the GeoJSON output file
  - Serves sites, stats, and dust layer data to frontend
  - Static PNG overlays served for dust heatmaps

Step 7: Frontend Display
  - React app fetches data from API on load
  - Leaflet map renders site polygons color-coded by risk
  - Sidebar allows filtering by risk level or permit status
  - SiteCard shows detailed info when a site is clicked
  - Dust dispersion overlay can be toggled by time period and pollutant

================================================================================
SECTION 8: WHAT THE PROJECT CANNOT DO
================================================================================

1. CANNOT measure actual air quality. It models dust contribution using
   physics approximations, but cannot tell you real PM10/PM2.5 levels.

2. CANNOT verify permits against real data. The VMC permit data is fake
   (20 hardcoded entries). Almost every site shows as "unregistered."

3. CANNOT detect construction in real-time. The pipeline uses historical
   imagery with a weeks-to-months delay from satellite revisit cycles.

4. CANNOT work for cities other than Vadodara without reconfiguration.
   The AOI bounding box, model training data, school/hospital coordinates,
   and permit data are all Vadodara-specific.

5. CANNOT run without a Google Earth Engine account. GEE access requires
   a Google Cloud project with Earth Engine API enabled. The backend crashes
   without GEE credentials even for basic API serving.

6. CANNOT provide ground-truth accuracy metrics. The model was trained on
   ~200 labeled examples (small dataset). False positive rates are known to
   be significant, especially near riverbeds and highways.

7. CANNOT attribute health impacts. The "population exposed" metric uses a
   crude heuristic (areaM2 × 0.012), not census data or epidemiological
   models.

8. CANNOT track individual sites over time. Each pipeline run is a snapshot.
   There is no temporal tracking, change history, or site status updates.

9. CANNOT send alerts or notifications. There is no email, SMS, or push
   notification system. Data is only viewable through the web interface.

10. CANNOT be deployed publicly as-is. The backend requires GEE credentials
    that cannot be shared publicly, and the pipeline is run manually.

   court-formatted documentation with satellite images, detection dates,
